0.1 Product Understanding

0.1.1 Core Product Vision

Based on the prompt, the Blitzy platform understands that the new product is a Meeting Copilot — a locally-hosted, browser-first web application with Telegram integration designed specifically for marketing analytical meetings. The product captures live audio from a browser microphone, performs real-time speech-to-text transcription via an AI provider, generates incremental summaries, extracts structured insights (decisions, action items, risks, open questions), enables conversational Q&A with the meeting context, and delivers post-meeting reports through both the browser UI and a Telegram bot.

Functional Requirements:

  • Real-Time Transcription (Live Transcript): Capture microphone audio in the browser via getUserMedia() and stream it to an AI STT provider (OpenAI Realtime API via WebRTC as primary mode), displaying timestamped, speaker-attributed transcript segments in the UI with sub-5-second latency
  • Incremental Live Summary: Generate and update a Markdown summary snapshot every 30–60 seconds by sending transcript deltas to an LLM for structuring, displayed as a rolling live summary panel
  • Structured Insight Extraction: Continuously identify and deduplicate four categories — Decisions, Action Items (with optional owner and due date), Risks (with optional severity), and Open Questions — using LLM-based delta processing with hash-based deduplication and similarity matching
  • Conversational Copilot (Ask Agent): Accept free-text questions from the user during the meeting (e.g., "Суммируй последние 5 минут", "Какие KPI мы упомянули?") and return contextual answers by querying the current meeting state via the LLM
  • Post-Meeting Report Generation: Upon stopping the meeting, compile a final structured minutes document in Markdown/HTML containing all transcript, summary, decisions, actions, risks, and open questions
  • Telegram Bot Integration: Provide a Telegram bot operating in long-polling mode supporting commands (/start_meeting, /stop_meeting, /summary, /actions, /decisions, /ask) and auto-posting meeting results to a configured chat/channel
  • Meeting CRUD: Create, retrieve, stop, and export meetings via REST API endpoints
  • Export Capabilities: Download meeting reports in Markdown format; optionally HTML

Non-Functional Requirements:

  • Latency: UI updates every 1–5 seconds; summary refresh every 30–60 seconds
  • Reliability: Auto-save meeting state every 10–30 seconds; graceful degradation if connection drops
  • Privacy & Confidentiality: Do not persist raw audio by default; store only transcript and reports; provide "delete meeting" capability for local data removal
  • Cross-Platform: Must run on Windows (WSL2 + Ubuntu) and macOS natively
  • Minimal Installation: Single backend process (Python) + browser; no OS-level plugins required for MVP
  • Security: API keys stored server-side only; ephemeral tokens issued to the browser; Telegram bot restricted via user/chat whitelist

Implicit Requirements Surfaced:

  • WebSocket server for real-time bidirectional UI-backend communication
  • Meeting state persistence layer (SQLite for local-first, upgrade path to PostgreSQL)
  • AI provider abstraction layer to support switching between OpenAI and Anthropic Claude
  • CORS configuration for localhost development
  • Environment variable management for sensitive configuration (API keys, bot tokens)
  • Structured logging for debugging real-time audio pipelines
  • Graceful WebSocket reconnection handling in the frontend

For transcription-only sessions, <cite index="41-5,41-6">you can use the Realtime API for transcription-only use cases, either with input from a microphone or from a file — for example, to generate subtitles or transcripts in real-time.</cite> <cite index="41-9">To use the Realtime API for transcription, you need to create a transcription session, connecting via WebSockets or WebRTC.</cite> The transcription session supports models including gpt-4o-transcribe and whisper-1, with built-in server-side VAD (Voice Activity Detection).

0.2.2 Architecture Pattern Research

Recommended Architecture: Event-Driven Local Orchestrator with Direct WebRTC STT

The chosen architecture follows a three-tier pattern optimized for real-time audio processing:

  • Tier 1 — Browser (React SPA): Captures microphone audio, establishes WebRTC peer connection to OpenAI Realtime API for transcription, renders live UI, communicates with backend via WebSocket
  • Tier 2 — Local Backend (FastAPI): Issues ephemeral tokens, manages meeting state, runs incremental LLM processing for insight extraction, serves REST API, bridges to Telegram
  • Tier 3 — AI Providers (External): OpenAI Realtime API for transcription (WebRTC), OpenAI/Anthropic for structuring and Q&A (REST API)

Design Patterns Identified:

  • Event Sourcing for Meeting State: Transcript segments are immutable events; derived state (summary, decisions, actions) is computed from the event log
  • Delta Processing Pipeline: Every N seconds, the backend computes deltas from new transcript segments and updates the aggregate state, avoiding full reprocessing
  • Provider Abstraction (Strategy Pattern): AI provider interfaces allow swapping between OpenAI and Anthropic without changing business logic
  • Observer Pattern for WebSocket Broadcasting: Meeting state changes are pushed to all connected WebSocket clients (browser UI) as delta or snapshot events

Common Pitfalls Identified:

  • Audio pipeline latency accumulation: Buffering audio on the server side before forwarding to STT creates compounding delays — Mode A (direct WebRTC) avoids this
  • Deduplication drift: Without stable IDs based on normalized text hashing, incremental LLM updates produce duplicate actions/decisions
  • WebSocket reconnection gaps: Browser tab suspension or network blips require reconnection with state resync, not just reconnect
  • Ephemeral token expiration: OpenAI ephemeral tokens are short-lived; the frontend must proactively refresh before expiry

0.3.2 Architecture Pattern

Overall Pattern: Event-Driven Local Orchestrator with Direct-to-Provider WebRTC

The system employs a hybrid architecture where the browser handles real-time audio capture and direct AI communication (WebRTC transcription), while the backend orchestrates state management, LLM-based structuring, integrations, and persistence.

Component Interaction Model:

  • Audio Flow: Browser microphone → WebRTC → OpenAI Realtime API → transcript events → browser UI → WebSocket → backend state manager
  • Insight Flow: Backend state manager receives transcript deltas → LLM service processes deltas → produces summary/actions/decisions/risks/questions → state persisted → delta broadcast to UI via WebSocket
  • Command Flow: User types question in chat → WebSocket user.question event → backend LLM service generates answer from meeting context → bot.answer event → UI displays response
  • Telegram Flow: Telegram command received via long polling → processed by Telegram bot bridge → queries meeting state → sends formatted response back to Telegram chat

Data Flow Architecture:

  • Transcript segments are the immutable source of truth, appended incrementally
  • Derived state (summaries, insights) is computed via periodic delta processing
  • WebSocket events carry either full state snapshots (meeting.state) or incremental updates (meeting.delta)
  • All state changes are persisted to SQLite with configurable auto-save intervals (10–30 seconds)

Security Architecture:

  • OpenAI API keys are stored exclusively on the backend in .env files, never exposed to the browser
  • The backend generates ephemeral tokens via POST /v1/realtime/client_secrets for the browser's WebRTC session
  • Telegram bot token is server-side only; bot commands are gated by a user/chat whitelist
  • Local deployment assumes trusted localhost network; no additional auth layer for MVP
  • CORS configured to allow only localhost origins

0.3.3 Integration Points

OpenAI Realtime API (Transcription via WebRTC):

  • Protocol: WebRTC peer connection from browser to OpenAI
  • Authentication: Ephemeral key generated by backend via POST https://api.openai.com/v1/realtime/client_secrets
  • Session type: "transcription" with model gpt-4o-transcribe
  • Events consumed: conversation.item.input_audio_transcription.delta, conversation.item.input_audio_transcription.completed
  • Configuration: Server VAD enabled, noise reduction near_field, 24kHz PCM input

OpenAI / Anthropic Chat Completions (Structuring & Q&A):

  • Protocol: HTTPS REST API from backend
  • Authentication: API key stored in backend .env
  • Operations: Incremental structuring of transcript deltas, answering user questions, generating final reports
  • Data format: JSON request/response with structured prompts and meeting context
  • Provider switching: Configurable via environment variable AI_PROVIDER=openai|anthropic

Telegram Bot API:

  • Protocol: HTTPS long-polling via python-telegram-bot library
  • Authentication: BOT_TOKEN from .env
  • Operations: Command handling (/start_meeting, /stop_meeting, /summary, /actions, /decisions, /ask), auto-posting meeting results
  • Target: Configurable TELEGRAM_DEFAULT_CHAT_ID for auto-post destination
  • Security: Whitelist-based command authorization

WebSocket (UI ↔ Backend):

  • Protocol: ws://localhost:PORT/ws/meetings/{meeting_id}
  • Events (Client → Server): user.question, meeting.command
  • Events (Server → Client): meeting.state, meeting.delta, bot.answer, bot.flag
  • Reconnection: Client implements exponential backoff with state resync on reconnect

0.5 Implementation Specifications

0.5 Implementation Specifications

0.5.1 Core Components To Implement

Component A: Meeting State Manager

  • Purpose: Central orchestration engine that maintains the complete meeting state, processes transcript deltas at configurable intervals (30–60 seconds), triggers LLM-based insight extraction, deduplicates results, and broadcasts updates via WebSocket
  • Location: backend/src/services/state_manager.py
  • Key interfaces: process_transcript_delta(meeting_id, segments), get_current_state(meeting_id), trigger_incremental_update(meeting_id), finalize_meeting(meeting_id)
  • Dependencies: AI Provider Service, Deduplication Service, Persistence Layer, WebSocket broadcaster

Component B: AI Provider Service (Abstract + Implementations)

  • Purpose: Provides a provider-agnostic interface for all AI operations — ephemeral token generation, transcript structuring, Q&A answering, and final report generation — enabling seamless switching between OpenAI and Anthropic
  • Location: backend/src/services/ai/provider_base.py, backend/src/services/ai/openai_provider.py, backend/src/services/ai/anthropic_provider.py
  • Key interfaces: generate_ephemeral_token(session_config), update_state(transcript_delta, current_state) -> state_delta, answer(question, meeting_state) -> str, final_report(full_transcript, state) -> str
  • Dependencies: OpenAI Python SDK, Anthropic Python SDK, Prompt Templates

Component C: WebRTC Audio Pipeline (Frontend)

  • Purpose: Manages browser microphone capture via getUserMedia(), establishes and maintains WebRTC peer connection with OpenAI Realtime API, handles transcript events and feeds them to the meeting state pipeline
  • Location: frontend/src/hooks/useWebRTC.ts, frontend/src/services/webrtc.ts
  • Key interfaces: initializeSession(ephemeralToken), startCapture(), stopCapture(), onTranscriptDelta(callback), onTranscriptCompleted(callback)
  • Dependencies: Browser WebRTC APIs, Backend token endpoint

Component D: WebSocket Communication Layer

  • Purpose: Bi-directional real-time communication between frontend and backend for state synchronization, user commands, and copilot Q&A
  • Location: backend/src/api/websocket.py (server), frontend/src/hooks/useWebSocket.ts (client)
  • Key interfaces (Server): handle_connection(meeting_id), broadcast_delta(meeting_id, delta), broadcast_state(meeting_id, state)
  • Key interfaces (Client): connect(meetingId), sendQuestion(question), onDelta(callback), onAnswer(callback)
  • Dependencies: FastAPI WebSocket, Starlette WebSocket utilities

Component E: Telegram Bot Bridge

  • Purpose: Provides Telegram integration for remote meeting control and result distribution via long-polling command handling and automated message posting
  • Location: backend/src/integrations/telegram/bot.py, backend/src/integrations/telegram/commands.py
  • Key interfaces: start_bot(), stop_bot(), post_meeting_results(chat_id, meeting_state), command handlers for /start_meeting, /stop_meeting, /summary, /actions, /decisions, /ask
  • Dependencies: python-telegram-bot library, Meeting Service, AI Provider Service

Component F: Deduplication Engine

  • Purpose: Prevents duplicate entries in actions, decisions, risks, and open questions across incremental LLM updates using text normalization, hash-based identity, and similarity matching
  • Location: backend/src/services/deduplication.py
  • Key interfaces: normalize_text(text) -> str, compute_stable_id(normalized_text) -> str, is_duplicate(new_item, existing_items, threshold) -> bool, merge_items(existing, new) -> list
  • Dependencies: hashlib (stdlib), optional rapidfuzz for fuzzy matching

Component G: Export Service

  • Purpose: Generates structured meeting reports in Markdown and HTML formats from the finalized meeting state
  • Location: backend/src/services/export_service.py
  • Key interfaces: generate_markdown(meeting_state) -> str, generate_html(meeting_state) -> str
  • Dependencies: Jinja2 templates, Meeting State models

0.5.2 Data Models And Schemas

Core Entity Definitions:

Meeting:
  - meeting_id: str (UUID)
  - title: str
  - started_at: datetime
  - ended_at: datetime | None
  - status: enum (active, stopped)
TranscriptSegment:
  - segment_id: str (UUID)
  - meeting_id: str (FK)
  - ts_start: float (seconds from meeting start)
  - ts_end: float
  - speaker: str | None
  - text: str
  - confidence: float | None
LiveSummarySnapshot:
  - snapshot_id: str (UUID)
  - meeting_id: str (FK)
  - timestamp: datetime
  - summary_md: str
Decision:
  - id: str (hash-based stable ID)
  - meeting_id: str (FK)
  - text: str
  - timestamp: datetime
  - confidence: float | None
  - related_segment_ids: list[str]
ActionItem:
  - id: str (hash-based stable ID)
  - meeting_id: str (FK)
  - text: str
  - owner: str | None
  - due_date: str | None
  - status: str (default: "draft")
Risk:
  - id: str (hash-based stable ID)
  - meeting_id: str (FK)
  - text: str
  - severity: str | None (low/medium/high)
  - timestamp: datetime
OpenQuestion:
  - id: str (hash-based stable ID)
  - meeting_id: str (FK)
  - text: str
  - timestamp: datetime

Relationship Map:

  • Meeting (1) → TranscriptSegment (many): Ordered by ts_start
  • Meeting (1) → LiveSummarySnapshot (many): Ordered by timestamp
  • Meeting (1) → Decision/ActionItem/Risk/OpenQuestion (many each): Linked by meeting_id
  • Decision → TranscriptSegment (many-to-many): Via related_segment_ids

Deduplication Rules:

  • Stable ID = SHA-256 hash of normalized text (lowercase, stripped punctuation, collapsed whitespace)
  • Similarity threshold: 0.85 (configurable) for fuzzy matching via Levenshtein distance
  • On conflict: Existing entry is preserved; new metadata (timestamp, confidence) merged
